Section 01

Problem Framing

Core Use Case

SCG's Senior Expert produces high-stakes legal work product — initial expert reports, deposition summaries, opposing expert rebuttal analyses, and governing code/standards compilations — in a specific, disciplined voice developed over years of practice. The labor bottleneck is not the judgment; it is the mechanical production work that must precede it: ingesting discovery documents, extracting key facts from depositions, locating and organizing applicable codes, and assembling a credible first draft. BlackBar offloads that bottleneck to an AI agent team so the Senior Expert can spend time on analysis, not assembly.

Primary Users

UserRole in the SystemInteraction Mode
Senior ExpertOwner. Defines voice, approves output, signs reports.CLI commands + file review + approval prompts
Case StaffMatter setup, document uploads, running intake.CLI + file drop to matter folder
Client / CounselReceives final output only. No system access.Delivered .docx or PDF via Clio

Success Criteria

Must achieve in v1 A first draft of any standard deliverable (report section, deposition summary, opposing expert analysis) reaches the Senior Expert in less than 30 minutes from document upload — with every factual claim traceable to a source file and line, every code citation verified against the loaded standard, and the draft voice indistinguishable from the Senior Expert's own prior work on first read.

Secondary criteria: no matter content ever leaves local file storage without explicit export, every draft version is retained, and every human approval is logged.

Section 02

Simplest Recommended Architecture

The One-Sentence Architecture

A Claude Code orchestrator session calls four specialized subagents in sequence — Intake, Research, Drafting, and QA — using local markdown files as shared memory, with a human approval checkpoint before any output leaves the system.

Why This Is Sufficient

SCG matters are sequential in nature: you intake, then research, then draft, then review. There is no need for parallel execution, a message queue, a vector database, or a web API in the first version. The document volumes per matter are in the tens to low hundreds of files — well within what file-based retrieval handles without infrastructure complexity. The agent count stays at four because each boundary maps to a genuine role change: document processor → analyst → writer → reviewer. Adding agents beyond these four would split work without adding capability.

Guiding principle Every component must justify its existence by either (a) producing a distinct artifact the Senior Expert can inspect, or (b) enforcing a hard constraint (confidentiality, citation traceability, voice fidelity) that the prior agent cannot self-enforce. If a proposed component does neither, it does not ship in v1.

Storage Pattern

All matter content lives in a structured local folder hierarchy. No cloud database, no vector store, no external API calls for matter data. Retrieval is file read + grep. This is intentional: it keeps the system auditable, discoverable by opposing counsel only if exported, and trivially backed up.

/blackbar/ ├── global/ │ ├── VOICE.md # Senior Expert's writing profile (the single most important file) │ ├── templates/ # Report structure templates by type │ └── examples/ # 3–5 sanitized prior reports as few-shot references │ ├── matters/ │ └── [MATTER-ID]/ # e.g., SCG-2026-014 │ ├── MATTER.md # Master context: parties, scope, deadlines, key facts │ ├── docs/ # Raw ingested documents (depositions, contracts, reports, photos) │ ├── standards/ # Building codes, development codes, consensus standards for this matter │ ├── research/ # Research outputs: depo summaries, opposing expert analysis, code extracts │ ├── drafts/ # Versioned drafts: draft_v01.md, draft_v02.md, etc. │ ├── approved/ # Final approved output ready for export │ └── APPROVAL_LOG.md # Timestamped record of every human approval action │ └── agents/ ├── CLAUDE.md # Global orchestrator context (loaded automatically by Claude Code) ├── intake.md # Intake agent system prompt ├── research.md # Research agent system prompt ├── drafting.md # Drafting agent system prompt └── qa.md # QA agent system prompt
Section 03

Agent Team Design

Four agents, each with a single clear responsibility. The main Claude Code session acts as the orchestrator — it does not produce work product itself; it sequences agents, manages human checkpoints, and handles handoff logic.

1
Intake Agent
Matter Setup & Document Ingestion

Receives raw matter materials, classifies every document, extracts key structured facts, and writes the master MATTER.md context file. This is the only agent that touches raw unprocessed input.

Purpose

Transform a folder of heterogeneous documents into a clean, structured matter brief that every downstream agent can trust.

Inputs
Retainer / engagement letter Complaint / petition Client brief Deposition transcripts Contracts & incident reports Opposing expert reports Photographs / site imagery Construction / permit docs
Outputs
MATTER.md Document index (type, filename, brief description) Preliminary scope summary
Tools
Read Write Bash (text extraction) Glob
Memory / Context

Reads global VOICE.md only to understand SCG's preferred terminology. Writes all outputs to /matters/[ID]/. No access to other matter folders.

Handoff Rule

Completes when MATTER.md is written and the document index is populated. Triggers human checkpoint #1. Does not proceed to Research until the Senior Expert confirms scope.

2
Research Agent
Analysis, Extraction & Standards Compilation

The deep reader. Given specific research tasks from the orchestrator, it extracts structured facts from depositions, deconstructs opposing expert methodology, and identifies applicable code and standards provisions — with pinpoint source citations on every finding.

Purpose

Produce a research package that makes the Drafting Agent's job a matter of organizing and writing, not searching and verifying.

Inputs
MATTER.md Document index Specific research queries (from orchestrator) Raw docs in /docs/ and /standards/
Outputs
Deposition summary (structured, page-line cited) Opposing expert analysis brief Applicable standards extract Citation registry (source → claim map)
Tools
Read Grep Write Bash (PDF text extraction)
Memory / Context

Reads MATTER.md and all files in the matter's /docs/ and /standards/ folders. Writes to /research/. No write access to drafts.

Handoff Rule

Completes when all requested research tasks have corresponding output files in /research/. Triggers human checkpoint #2. Does not draft — its output is exclusively factual extraction and analysis framing.

3
Drafting Agent
Work Product Generation in Senior Expert's Voice

The writer. Consumes the research package and produces a complete first draft of the requested deliverable — initial report, rebuttal section, or deposition summary — structured exactly as the Senior Expert would write it, in their voice.

Purpose

Produce a draft that requires refinement and expert judgment, not reconstruction. The Senior Expert should recognize their own voice and structure immediately.

Inputs
Research package from /research/ VOICE.md Applicable template from /global/templates/ 1–2 example reports from /global/examples/ Drafting task spec (from orchestrator)
Outputs
draft_v01.md in /drafts/ Inline citation tags [SOURCE: filename, p.X] Draft cover page and section headers Placeholder flags for Senior Expert judgment calls
Tools
Read Write
Memory / Context

Reads VOICE.md, applicable template, 1–2 example reports, and all files in /research/. Writes only to /drafts/. Never accesses raw source documents directly — all factual grounding comes pre-processed from Research.

Handoff Rule

Completes when a full draft is written to draft_v01.md. All unresolvable judgment calls are flagged inline with [EXPERT JUDGMENT REQUIRED: ...]. Passes draft to QA Agent automatically.

4
QA Agent
Citation Verification, Voice Check & Approval Prep

The final gate before human review. Verifies every factual claim in the draft against the source documents in the citation registry, checks voice consistency against VOICE.md, flags anything unverifiable, and produces a structured QA report for the Senior Expert.

Purpose

Ensure the Senior Expert never receives a draft with an unsupported fact or an inconsistent voice pass. Every flag is actionable, not generic.

Inputs
draft_v01.md Citation registry All source documents VOICE.md
Outputs
QA_REPORT.md (pass/flag/fail per claim) Annotated draft with inline flags Summary: N claims verified, M flagged, P unresolvable
Tools
Read Grep Write
Memory / Context

Full read access to the matter folder. Write access only to /research/QA_REPORT.md and the annotated draft copy. Never modifies the original draft.

Handoff Rule

Triggers human checkpoint #3. The Senior Expert reviews the annotated draft alongside the QA report, makes edits, and issues an explicit approval command. Only after approval does the orchestrator write to /approved/ and log the event in APPROVAL_LOG.md.

Section 04

Workflow Sequence

Twelve steps from matter intake to approved output. Steps marked with a human pause icon require explicit Senior Expert action before the orchestrator continues.

1
CASE STAFF

Matter Folder Created

Case staff creates a new matter directory using the standard naming convention (SCG-YYYY-NNN) and drops all available documents — retainer, complaint, depositions, opposing expert reports, relevant building codes, photographs — into /docs/ and /standards/ respectively. No renaming or pre-processing required.

2
ORCHESTRATOR

Matter Initialization

Senior Expert (or staff) runs: blackbar intake SCG-2026-014. The orchestrator spins up the Intake Agent, passes the matter folder path, and instructs it to produce MATTER.md and the document index.

3
INTAKE AGENT

Document Ingestion & Matter Brief

Intake Agent reads and classifies every file: extracts parties, alleged code violations, key dates, assignment scope, and outstanding unknowns. Writes structured MATTER.md. Flags any documents it cannot parse.

✓
HUMAN CHECKPOINT #1 — Scope Confirmation
Senior Expert reviews MATTER.md. Confirms scope, adds any missing context, explicitly approves to proceed. Required before Research begins.
4
ORCHESTRATOR

Research Task Generation

Based on confirmed MATTER.md and deliverable type (initial report / rebuttal / deposition summary), the orchestrator generates a specific research task list and passes it to the Research Agent. Task list is written to MATTER.md as a logged record.

5
RESEARCH AGENT

Deposition Extraction

For each deposition transcript, Research Agent extracts: key admissions, contradictions, relevant testimony by topic, and page/line citations. Outputs structured summary to /research/depo_[witness]_summary.md.

6
RESEARCH AGENT

Opposing Expert Analysis

For each opposing expert report, Research Agent extracts: stated opinions, methodology, code citations used, logical dependencies, and identifiable gaps or vulnerabilities. Outputs to /research/opposing_expert_[name]_analysis.md with every claim mapped back to a page and paragraph in the source report.

7
RESEARCH AGENT

Code & Standards Compilation

Identifies the applicable edition of each referenced building code, development code, or consensus standard. Extracts the specific sections at issue. Cross-references permit date with code edition in force at time of construction. Outputs citation registry to /research/citation_registry.md.

✓
HUMAN CHECKPOINT #2 — Research Review
Senior Expert reviews research outputs. Confirms the factual extraction is accurate and complete. May add supplemental context. Approves to proceed to drafting.
8
ORCHESTRATOR

Drafting Task Spec

Orchestrator composes a precise drafting instruction: deliverable type, section order from the applicable template, tone register (rebuttal vs. primary opinion), any specific arguments the Senior Expert flagged in research review, and the target length range.

9
DRAFTING AGENT

First Draft Generation

Drafting Agent loads VOICE.md, the applicable template, one or two example reports, and the complete research package. Produces draft_v01.md with inline citation tags, structured exactly as the Senior Expert would produce it. Every claim that requires expert judgment is flagged explicitly rather than fabricated.

10
QA AGENT

Citation Verification & Voice Check

QA Agent verifies every factual claim in the draft against source documents. Checks that each code citation matches the loaded text of the standard. Scores voice consistency against VOICE.md. Produces QA_REPORT.md and an annotated draft copy.

✓
HUMAN CHECKPOINT #3 — Senior Expert Review & Approval
Senior Expert reviews annotated draft alongside QA report. Makes edits directly in the draft file. Runs blackbar approve SCG-2026-014 v01. Orchestrator copies approved version to /approved/, logs timestamp and approver in APPROVAL_LOG.md. Work product is now export-ready.
11
ORCHESTRATOR

Export & Clio Handoff

On export command, orchestrator converts the approved markdown to the appropriate output format (Word .docx for counsel, PDF for submission). Optionally generates a cover memo. Export is a deliberate manual step — nothing leaves the matter folder automatically.

12
CASE STAFF

Matter Close & Archive

After matter completion, case staff runs blackbar archive SCG-2026-014. Orchestrator writes a final LESSONS.md file (what worked, what flagged, what to improve) and moves the matter folder to the archive directory. The voice examples library is optionally updated with a sanitized excerpt from this matter's approved output.

Section 05

Voice Preservation Strategy

The VOICE.md File (Most Critical Artifact)

VOICE.md is a living document maintained by the Senior Expert. It is the single most important file in the system. It is not a style guide — it is a behavioral profile written in first person that the Drafting Agent uses as a continuous reference while writing.

It contains six sections:

SectionWhat It CapturesExample Entry
Reasoning Structure How the Expert moves from facts to conclusions. The sequence of argument construction. "I always establish what the applicable standard required at the time of construction before addressing what was built. Never reverse this order."
Preferred Phrasing Recurrent phrases, preferred verbs, sentence openings the Expert uses. "I use 'the evidence establishes' not 'it appears that.' I use 'the 2018 IBC requires' not 'the code says.'"
Report Structure Required section order, section titles, what goes in each section. "Assignment section always precedes Documents Reviewed. Opinions are stated at the end, not introduced early."
Citation Discipline How sources are cited inline and in a references section. Format by document type. "Depositions: [Witness Last Name Depo., p. X, l. Y]. Codes: [IBC 2018 § X.X.X]. Expert reports: [Smith Report, p. X, ¶ Y]."
Prohibitions Things the Expert never writes. "Never use 'clearly' or 'obviously.' Never state a legal conclusion. Never use passive voice for opinions."
Judgment Flags Classes of statements that must be flagged for Expert judgment, not drafted by AI. "Any opinion about causation. Any quantification of damages. Any statement about what a party knew or intended."

Few-Shot Examples

The Drafting Agent loads one or two example reports from /global/examples/ that most closely match the current deliverable type (rebuttal report vs. initial report, code compliance vs. causation analysis). These are sanitized prior reports with party names and case-specific details redacted. They serve as structural and voice anchors — not templates to fill in, but reference patterns to match.

How Voice is Enforced at the QA Stage

The QA Agent performs a voice consistency check against four specific criteria from VOICE.md: prohibited terms, citation format, section order, and reasoning sequence. Any deviation is flagged with a specific reference to the VOICE.md rule it violates. The QA Agent does not silently rewrite for voice — it flags and surfaces for human correction. This is intentional: voice refinement over multiple iterations teaches the system, while silent rewriting teaches nothing.

Voice Improvement Loop

After each matter closes, the Senior Expert spends fifteen minutes reviewing the QA flags and deciding whether any represent systematic gaps in VOICE.md. If so, they update the file directly. Over the first ten to fifteen matters, VOICE.md will become a precision instrument that requires very few corrections per draft.

Section 06

Guardrails & QA

Confidentiality

RiskControl
Matter content exposed to cloud LLM providers BlackBar runs entirely on Claude Code locally. All file operations are local. No matter content is sent to any external API unless the Senior Expert explicitly uses a cloud-connected tool. Document this constraint prominently in the CLAUDE.md global context.
Cross-matter contamination Each agent is instantiated with access scoped to a single matter folder. The orchestrator passes matter-specific paths — never the parent /matters/ directory. Agents cannot glob across matters.
Inadvertent export before approval The export command requires an approved version to exist in /approved/. The orchestrator checks this before running conversion. No unapproved draft can be exported by command — staff would have to manually copy files to bypass this.
Discovery risk from AI process artifacts All draft files, QA reports, and research outputs are clearly labeled as attorney-work-product-adjacent. Consult with retained counsel on whether to apply work product labels. The system does not generate privilege logs — that is out of scope for v1.

Hallucination Control

LayerMechanism
Research → Drafting isolation The Drafting Agent has no direct access to raw source documents. It can only use what the Research Agent extracted and placed in /research/. This limits the surface area for fabrication: the Drafting Agent cannot invent a deposition quote that was never extracted.
Inline citation requirement The Drafting Agent's system prompt requires every factual claim to carry a [SOURCE: filename, location] tag. Any claim without a tag is automatically flagged by the QA Agent as unverifiable.
QA verification pass The QA Agent independently reads the source documents and checks each tagged claim. It cannot verify claims that have no source tag, and it flags any tag that does not match the quoted or paraphrased content.
Expert judgment flags The Drafting Agent is instructed to write [EXPERT JUDGMENT REQUIRED: {description}] in place of any opinion, causal inference, or damage quantification. These placeholders are never synthesized — they are always deferred to the Senior Expert.

Evidentiary Traceability

The citation registry produced by the Research Agent is the evidentiary spine of every deliverable. It maps each factual claim to a specific source document, page, and location. This registry is retained in the matter folder permanently. If a claim in a final report is ever challenged in deposition or cross-examination, the Senior Expert can trace it back to the source in seconds. The registry is never deleted — it is part of the matter record.

Version Control

Draft files are never overwritten. Each revision cycle creates a new file: draft_v01.md, draft_v02.md, etc. The APPROVAL_LOG.md records which version was approved, by whom, and at what timestamp. The orchestrator enforces this — write operations to /drafts/ always create new files with auto-incremented version numbers. Prior versions are read-only after a new version is created.

Hard Constraint The QA Agent must never achieve a "pass" status on any claim it cannot verify in the loaded source documents. A claim that cannot be verified must be flagged UNVERIFIABLE — not passed, not silently removed. The Senior Expert decides what to do with unverifiable claims. The system does not make that call.
Section 07

What to Postpone

The following capabilities are real and eventually valuable — but each adds architectural complexity that is not justified until the four-agent core is operating reliably across ten or more matters.

✗
Vector database / semantic search
File-based grep retrieval is sufficient for per-matter document volumes. A vector store adds infrastructure, maintenance, and a new failure mode. Revisit when a single matter routinely exceeds 200 documents or 500 pages of depositions.
✗
Clio API integration
Clio sync would automate matter creation and export. It also introduces an authenticated external dependency and a potential discovery surface. Manual export to Clio takes two minutes. Not worth the integration risk in v1.
✗
Parallel agent execution
Running Research sub-tasks in parallel (depositions, opposing expert, codes simultaneously) would cut processing time. The sequential model is simpler to debug and the time savings are modest relative to the added orchestration complexity in v1.
✗
Web search / external legal database integration
Connecting to Westlaw, ICC codes online, or ASTM portals adds authentication, rate limiting, and data provenance complexity. All standards should be loaded as local files in /standards/ in v1.
✗
Automated privilege logging
Generating privilege logs and work product designations requires legal counsel input on applicable doctrine. Out of scope for an AI system to decide. Mark as a future integration with retained counsel's review process.
✗
Multi-matter cross-referencing
The ability to surface findings from prior similar matters is genuinely useful — especially for recurring opposing experts and recurring code sections. Build this after the per-matter workflow is stable, not before.
✗
GUI / web interface
The CLI is the interface in v1. The Senior Expert is a power user; a web UI adds front-end development scope that is not on the critical path to getting the drafting quality right.
✗
Automated deposition question preparation
A useful future capability: generating suggested deposition questions based on the opposing expert analysis. Depends on the Research Agent's opposing expert output being proven reliable first.
Section 08

Example Matter: Harrison v. Ridgeline Commercial

Matter Brief — SCG-2026-014
Matter IDSCG-2026-014
ClientRidgeline Commercial Properties, LLC (Property Owner / Defendant)
ClaimPlaintiff Harrison alleges a slip and fall on an exterior stair at Ridgeline's commercial building. Claims the handrail system was non-compliant with the IBC at time of construction, causing lack of adequate support during descent.
Opposing ExpertSmith Engineering Associates — Report signed by Dr. A. Smith, PE. Opinion: the installed handrail violated IBC 2018 §1014 (graspability and height requirements).
AssignmentRetained to rebut Smith's report. Analyze whether the handrail met the applicable code at the time the construction permit was issued (permit date: March 2019).
DeliverableRebuttal Expert Report
Deadline45 days from retainer

Flow Through the Architecture

Step 1–2 · Intake

Case staff drops nine files into /matters/SCG-2026-014/docs/: retainer letter, complaint, Ridgeline's answer, Smith Engineering report, deposition of Harrison (208 pages), deposition of property manager Rodriguez (64 pages), site photographs (22 files), construction permit, and building inspection certificate. The IBC 2018 PDF is placed in /standards/.

The Intake Agent classifies all nine documents, extracts the parties, identifies the specific code sections Smith cited (§1014.3 graspability profile, §1014.4 height), notes the permit date, and writes MATTER.md. The document index lists all files with type and a one-sentence description. Intake flags one unknown: the architectural drawings referenced in the Smith report are not in the document set.

Checkpoint #1: Senior Expert reviews MATTER.md, confirms scope, notes that architectural drawings should be requested, and approves to proceed. The missing drawings are logged as an open item in MATTER.md.

Steps 4–7 · Research

The orchestrator generates four research tasks: (1) summarize Harrison deposition for facts bearing on the incident and handrail interaction; (2) summarize Rodriguez deposition for property maintenance and construction history; (3) extract and analyze Smith's IBC citations and methodology; (4) pull the full text of IBC 2018 §1014 and identify the code in force in March 2019 by jurisdiction.

Research Agent outputs: depo_harrison_summary.md (Harrison testified he did not grip the handrail at all — flagged as significant); depo_rodriguez_summary.md (Rodriguez confirms handrail installed per approved drawings); opposing_expert_smith_analysis.md (Smith's graspability opinion relies on §1014.3 Table — Research Agent notes Smith cited the 2018 edition but the permit predates its adoption in this jurisdiction by 14 months); citation_registry.md (all source mappings).

Checkpoint #2: Senior Expert reviews. Notes that the jurisdiction adoption date discrepancy is the core of the rebuttal. Approves research. Adds a note: "Lead with the edition-in-force argument. Smith's entire opinion collapses if the 2015 IBC applies."

Steps 8–9 · Drafting

The orchestrator spec instructs the Drafting Agent to produce a rebuttal report using the rebuttal template, leading with the code-edition-in-force argument, following the citation format in VOICE.md, and flagging the causation link between handrail geometry and the fall for Expert judgment. The Drafting Agent loads the rebuttal template, two prior rebuttal reports from /global/examples/, and writes draft_v01.md. The draft's Assignment section cites the retainer; the Documents Reviewed section lists all nine source documents; the Analysis section opens with the edition-in-force argument, citing [IBC 2018 Adoption History — Jurisdiction X, p.3] and [Harrison Permit, p.1]; the rebuttal of Smith's methodology follows; and the Opinions section contains three opinions and one [EXPERT JUDGMENT REQUIRED: causation link between handrail geometry and plaintiff's fall mechanics] placeholder.

Step 10 · QA

QA Agent verifies all 23 inline citations in the draft. 22 verify cleanly. One citation — a claim that "the 2015 IBC was adopted in [Jurisdiction X] effective January 2017" — cannot be verified against any loaded source document. QA flags it as UNVERIFIABLE. Voice check: two instances of "clearly" detected (prohibited per VOICE.md), flagged. Report structure check: section order matches rebuttal template.

Checkpoint #3: Senior Expert reviews QA report and annotated draft. Fixes the two "clearly" instances. Looks up the jurisdiction adoption date, confirms it, and adds the source document to /standards/. Re-runs QA on that single claim (QA Agent can be invoked for spot checks). All claims now verified. Approves: blackbar approve SCG-2026-014 v01. Draft copied to /approved/, approval logged.

Steps 11–12 · Export & Archive

Senior Expert runs export. System converts draft_v01.md to Word format. Senior Expert formats cover page, adds signature block, delivers to counsel via Clio. Matter archived on close. The pattern — opponent relied on wrong code edition — is noted in LESSONS.md as a recurring rebuttal pattern to watch for in future matters.

Outcome The Senior Expert spent time on two things: confirming scope, and the causation judgment call. Every other component — document ingestion, deposition extraction, code research, opposing expert deconstruction, first draft — was produced by the agent team. The critical insight (wrong code edition) was surfaced by Research, not stumbled on in the Senior Expert's own reading.
Section 09

Final Recommendation

Build This First

Start with the VOICE.md file and two to three sanitized example reports. This is the most important investment in the system and costs nothing but the Senior Expert's time. Every hour spent on VOICE.md returns multiples in reduced revision cycles on every matter that follows.

Then build the four agents in order: Intake first (simplest, most observable), Research second (highest value, most testable), Drafting third (most dependent on VOICE.md quality), QA last (requires the others to function to be useful).

Run three matters sequentially before adjusting anything. The first will reveal what VOICE.md is missing. The second will reveal what the citation registry needs. The third should produce a draft the Senior Expert considers genuinely useful as a starting point.

  • Week 1: Build VOICE.md, collect example reports, create folder structure, write agent system prompts.
  • Week 2: Build and test Intake Agent on a closed matter with known documents.
  • Week 3: Build and test Research Agent. Validate citation registry accuracy against source documents manually.
  • Week 4: Build Drafting Agent. Run end-to-end on a closed matter. Compare output to the actual report filed.
  • Week 5: Build QA Agent. Validate that every hallucination and voice deviation from Week 4's test is caught.
  • Week 6–8: Run on first live matter. Refine VOICE.md after each checkpoint. Do not add features until the voice is right.

The architecture is deliberately small. Four agents, a file system, and a voice profile. The complexity that matters is in the quality of VOICE.md and the precision of the citation discipline — both of which are the Senior Expert's domain, not the system's. BlackBar's job is to do the mechanical work reliably and traceably so that the Senior Expert's judgment is applied where it is irreplaceable.
